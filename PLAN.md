# Merch Maverick Frontend Audit And Implementation Roadmap

## Summary
- The real product is a B2B factory-direct custom merchandise funnel, not the AI merch-design app described in [audit.md](</Users/faiazyen/Desktop/Merch Maverick Codex/audit.md:9>). The live site and local repo match each other; the audit mostly reviewed a different public surface.
- Build health is deceptively fine: `npm run lint` and `npm run build` both pass. The problems are product truthfulness, IA discipline, and frontend system quality, not compilation.
- Public interface changes that should be treated as canonical: keep `/`, `/quote`, `/pricing`, `/about`, `/contact`, `/testimonials`, `/sustainability`, `/solutions/[vertical]`, `/sign-in`, `/portal`; remove or gate `/admin` and `/preview`; collapse legacy aliases `/signin`, `/sign-up`, `/corporate`, `/events`, `/fitness`, `/hospitality`, `/industrial` once redirects are no longer needed.
- Assumption: [audit.md](</Users/faiazyen/Desktop/Merch Maverick Codex/audit.md:9>) is stale or based on an older deployment. Current code and [https://merch-maverick.vercel.app/](https://merch-maverick.vercel.app/) are the source of truth.

## 1. Product understanding
- This product is a premium B2B merch manufacturer lead-gen site with light software veneer: marketing site, instant quote wizard, sign-in, and an early client portal. The core promise is factory ownership, 3D approvals, better margins, and category-specific merch programs, not consumer DIY design.
- The target user is procurement/ops/marketing for hospitality groups, corporate teams, fitness brands, industrial companies, events, and creators. The vertical strategy is explicit in [HeroSection](</Users/faiazyen/Desktop/Merch Maverick Codex/src/components/home/HeroSection.tsx:18>), [VerticalsSection](</Users/faiazyen/Desktop/Merch Maverick Codex/src/components/home/VerticalsSection.tsx:8>), and [solutionData](</Users/faiazyen/Desktop/Merch Maverick Codex/src/lib/solutionData.ts:1>).
- The intended funnel is `home -> vertical proof or pricing -> /quote -> sign-in -> /portal -> future reorders/tracking`. That flow is coherent in concept.
- What is weak today: the portal and admin story are ahead of the shipped product; multiple pages imply operational tooling that does not exist yet; several trust claims are presented as live capabilities when they are placeholder content.

## 3. Component/system map
- Real reusable primitives exist, but they are thin: [MaverickLogo](</Users/faiazyen/Desktop/Merch Maverick Codex/src/components/branding/MaverickLogo.tsx:1>), [Button](</Users/faiazyen/Desktop/Merch Maverick Codex/src/components/ui/Button.tsx:1>), [Input](</Users/faiazyen/Desktop/Merch Maverick Codex/src/components/ui/input.tsx:1>), [Signup1](</Users/faiazyen/Desktop/Merch Maverick Codex/src/components/ui/signup-1.tsx:62>), [QuoteTool](</Users/faiazyen/Desktop/Merch Maverick Codex/src/components/quote/QuoteTool.tsx:46>), [SolutionPageLayout](</Users/faiazyen/Desktop/Merch Maverick Codex/src/components/verticals/SolutionPageLayout.tsx:70>).

## 6. Recommended implementation phases
1. `Reality alignment`: fix quote/contact submission truth, remove or gate `/admin` and `/preview`, rewrite portal/home claims so marketing matches shipped capability.
2. `IA and conversion pass`: lock canonical routes, remove auth/vertical alias clutter, fix CTA destinations, align nav/footer naming, add missing desktop contact/how-it-works paths.
3. `Systemization`: extract shared marketing primitives, standardize CTA/button semantics, centralize card/metric/panel patterns, and delete dead components.
4. `Portal and auth hardening`: decide whether `/portal` remains a lightweight account shell or becomes a real order surface; then shape Supabase-backed data contracts around that decision.
5. `Performance, accessibility, SEO`: convert static client components back to server where possible, replace CSS font import with `next/font`, fix interactive semantics, add missing metadata, and document the repo properly.

## Validation
- Implementation should be accepted only if: the primary funnel submits real data, no public route markets unshipped capabilities as live, `/admin` and `/preview` are no longer public product pages, canonical routes are reduced, `Link`/`button` nesting is removed, and lint/build still pass.
